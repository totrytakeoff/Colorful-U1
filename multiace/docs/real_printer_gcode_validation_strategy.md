# 实机 G-code 校验与 Web 发送策略

日期：2026-06-08

## 2026-06-30 route-select 命令名与 P3 风扇过滤

实机打印 `竖着.gcode` 时出现两类日志：

```text
Unknown command:"COLORFUL_U1"
M106: Unsupported fan ID: 3
extruder below minimum temp, temperature: 169.75
```

定位结论：

- 后处理器写入了 `COLORFUL_U1_ROUTE_SELECT ...`。
- Snapmaker/Klipper 的 G-code parser 按 `([A-Z_]+|[A-Z*/])` 拆命令，
  `COLORFUL_U1_ROUTE_SELECT` 会被拆成命令 `COLORFUL_U1`，因此注册
  `COLORFUL_U1_ROUTE_SELECT` 没有效果。
- route-select 本应执行真实 head 选择和可选 `ACE_SWAP_HEAD`。它失效后，切片
  G-code 仍继续执行换色后的冷却/挤出段，可能在已经被 `M104 S70 ... ;cooldown`
  降温的喷嘴上挤出，最终触发 `min_extrude_temp=170` 保护。
- `M106 P3 S200` 来自 Snapmaker Orca 的 `filament start gcode` 段；同一段前后
  已经有 `M106 P2 ...` 和默认 `M106 ...`。当前实机 Klipper 不支持 fan id `P3`，
  因此会持续报 `Unsupported fan ID: 3`。

修复规则：

- 新生成的 object-aware route 命令统一改为 `COLORFUL_ROUTE_SELECT`，避免命令名
  中间出现数字。
- Klipper 侧保留 `COLORFUL_U1` legacy shim，用于识别旧文件中的
  `COLORFUL_U1_ROUTE_SELECT` 并转发到同一套 route-select 逻辑；旧文件仍建议重新
  后处理生成。
- 后处理器继续能解析旧 `COLORFUL_U1_ROUTE_SELECT`，用于去重/兼容历史文件。
- 后处理器把 `M106 P3 ...` 改写为注释：

```gcode
; COLORFUL_FILTERED_UNSUPPORTED_FAN M106 P3 S200
```

选择过滤而不是映射的原因：

- 当前文件中已经存在 `M106 P2 ...`，强行把 P3 映射到 P2 可能改变冷却语义。
- P3 在当前固件上没有可执行对象，裸发送只会制造错误日志。
- 注释保留原始命令，便于后续如果确认 P3 对应真实硬件通道，再显式实现映射。

后续所有实机 G-code 回归必须断言：

- 不再输出 `COLORFUL_U1_ROUTE_SELECT`。
- 输出 `COLORFUL_ROUTE_SELECT`。
- `M106 P3 ...` 不再作为可执行 G-code 进入最终文件。

背景：在 Snapmaker U1 实机 `192.168.1.38` 上，用户通过
`http://192.168.1.38/multiace/` Web 页面上传并发送打印时，连续触发
Z 轴相关报错；同类任务通过官方切片软件直接发送可以正常打印。

本文记录本次问题的定位结论，并把后续必须落地的 Web 校验规则和实机优化策略
固化为工程要求。

## 2026-06-08 dry-run 更新：真实 U1 G-code route-plan 验证

本轮 dry-run 使用真实 Snapmaker Orca U1 切片：

```text
摆摊提示牌001_PLA_1h21m.gcode
generated by Snapmaker Orca 2.3.1
size: 4.3 MB
```

该文件不是 Bambu/P1S blocker，而是暴露了 route-plan rewrite 的边界问题。
文件中存在以下 no-op toolchange：

```gcode
; Change Tool0 -> Tool0 (layer -1)
T0
```

route plan 正确生成的真实事件序列为：

```text
[0, 3, 2, 1, 2, 3]
```

旧 post-processor 会把 `Change Tool0 -> Tool0` 后的裸 `T0` 当成一次新的
`tool_select` event 消费，导致 route cursor 错位，并在后续 rewrite 阶段报：

```text
route plan missing tool_select event commands for T0
```

修复规则：

- `Change ToolX -> ToolX` 视为 no-op marker。
- no-op marker 不消费 route plan event，不设置新的 pending route tool。
- no-op 后的裸 `Tn` 只按当前 route tool 对应的真实 head 重写。
- 只有真实 `Change ToolX -> ToolY` 且 `X != Y` 时，才由后续裸 `Tn` 消费下一个
  `tool_select` event。

验证结果：

- 同一份真实 G-code 在 dry-run 中完成
  `route-plan/preview -> route-plan/remap -> route-plan/validate -> route-plan/print`。
- 手动 remap T0-T3 到 ACE slot0-slot3 后，route validate 返回 ok。
- 最终 print job `done 100%`，`error=None`。
- 最终上传文件中实际 ACE swap 顺序为 slot `0 -> 3 -> 2 -> 1 -> 2 -> 3`。

浏览器 dry-run 验证结果：

- 上传真实 G-code 后，UI 显示 `需要手动映射 T1, T2, T3`，发送按钮保持禁用。
- source 候选卡显示完整 configured loadout；ACE slot 显示 `可用 · ready`，
  unavailable native source 显示原因并禁用。
- 选择 T0-T3 source 时不触发 `/api/route-plan/remap`。
- 选齐 source 后 Mapping 状态显示 `待应用`，不会继续显示旧 resolver 错误。
- 点击“应用映射”后只触发 1 次 `/api/route-plan/remap`，随后自动 validate。
- validate 通过后 Mapping/Route validate/Source state 为 `ok/ok/clean`，
  `发送打印`按钮解锁。
- 浏览器 console 无 warning/error。

后续样本库要求：

- 将 no-op `Change ToolX -> ToolX` 作为合法 U1 G-code 样本类别。
- 将真实切片的 marker 序列最小化为固定 dry-run 回归样本。
- 对所有 route-plan rewrite 错误保留原始 marker 行号、route event index 和
  当前 cursor 位置，避免错误信息只显示笼统的 `commands for Tn`。

## 实机现象

最近失败集中在同一份文件：

```text
六边拉桥_PLA_39m12s.gcode
```

Klipper 日志中最近三次失败时间：

```text
2026-06-06 14:14:35
2026-06-06 14:15:17
2026-06-06 14:19:28
```

核心报错一致：

```text
Must home Z axis first: 227.200 300.500 205.020
```

失败行：

```text
current_line_gcode=G380 S2 Z-25 F300 ;
```

报错前还出现以下 U1 Klipper 当前不支持或不兼容的命令：

```text
Unknown command:"M486"
Unknown command:"M201"
Unknown command:"M203"
Unknown command:"M205"
Unknown command:"M710"
Unknown command:"M290"
Unknown command:"M17"
Unknown command:"G380"
```

上传文件头部显示：

```gcode
; generated by Snapmaker Orca 2.3.1
;===== machine: P1S ========================
```

结论：Web 发送路径接受并直接打印了一份 Bambu/P1S 风格 G-code。该启动段不适配
U1 的 Klipper 环境，导致在 Z 未正确归零前执行 `G380` 相关动作并触发 hard
error。

## 根因判断

这不是 Z 轴硬件故障，也不是普通打印路径中的 Z 移动越界。

根因是 Web preflight 当前只关注 native/ACE 映射和 G-code rewrite，没有对
“这份 G-code 是否属于当前 U1 机型/当前固件可执行方言”做硬性校验。官方软件
直接发送正常，是因为它使用官方 U1 发送路径，或者在发送前完成了机型模板、
打印任务参数和固件命令集适配。

Web 路径当前行为：

```text
用户选择 G-code
-> /api/preflight
-> native/ACE source map
-> rewrite / inject auto-load
-> Moonraker /server/files/upload print=true
```

缺失的安全门：

```text
G-code 机型签名校验
G-code 方言/危险命令校验
启动段 home 顺序校验
官方任务参数与 Web 发送参数差异校验
```

## 必须新增的校验规则

### 1. 机型签名 blocker

Web preflight 必须扫描文件头部和 slicer 元数据。发现明显非 U1 目标机型时，
直接拒绝发送打印。

第一批 hard blocker：

```text
;===== machine: P1S
;===== machine: X1
;===== machine: X1C
;===== machine: A1
;===== machine: A1 mini
Bambu
```

错误提示应明确：

```text
当前 G-code 不是 Snapmaker U1 兼容文件。请使用 U1 配置重新切片后再通过
multiACE Web 上传。
```

### 2. 危险命令 blocker

发现以下命令时，默认拒绝 Web 发送。后续若确认某些命令在 U1 固件上安全，
再从 blocker 降级为 warning。

```text
G380
M710
M290
M960
M975
M970
M974
M620
M621
M620.1
M620.11
M628
M629
```

理由：

- `G380` 在本次事故中直接触发 `Must home Z axis first`。
- `M620/M621/M620.*` 是 AMS/Bambu 风格换料语义，不应在 U1 Web 发送路径中
  裸执行。
- `M970/M974/M975/M960` 属于机型特定检测/灯光/振动相关命令，不能假定 U1
  Klipper 支持。

### 3. 启动段 home 顺序校验

Web preflight 需要检查执行块前段，至少覆盖前 500 条非注释 G-code。

hard blocker 条件：

```text
在 G28 或 G28 Z 之前出现绝对 Z 移动
在 Z 未 homed 前出现 G38/G380/探针/校准类 Z 动作
在明显未完成 home 的状态下出现负 Y 擦嘴/丢料路径
```

本次样例中的失败模式：

```gcode
G91
G380 S2 Z30 F300
G380 S2 Z-25 F300
```

这类逻辑即使在源机型上合法，也不能直接交给 U1 Web 路径。

### 4. 已知无害 unknown command 不应掩盖 blocker

某些命令如 `M486` 可能只是对象标记，单独出现时可以 warning。但是只要同一
文件出现机型签名 blocker 或危险命令 blocker，就必须拒绝发送，不能因为
“Unknown command 可忽略”而继续打印。

### 5. 校验结果进入 source map

每次 preflight 的 `source_map` 或旁路 artifact 应记录：

```text
validation.version
validation.machine_signature
validation.blockers[]
validation.warnings[]
validation.scanned_header_lines
validation.scanned_command_lines
```

这样实机日志只看到 Klipper 报错时，也能回溯 Web 在发送前看到过什么。

## Web UI 行为要求

校验失败时：

- 禁用发送打印按钮。
- 显示第一个 hard blocker 的人类可读原因。
- 展开详情时显示命令、行号、规则名。
- 不提供“忽略并打印”按钮，除非后续实现专门的开发者调试模式。

示例：

```text
无法发送打印：检测到 P1S/Bambu 启动 G-code，不兼容 Snapmaker U1 Web 发送路径。
第 42 行：;===== machine: P1S
第 67 行：G380 S2 Z-25 F300
请使用 U1 配置重新切片。
```

## 实机优化策略

### 1. Web 发送路径先做稳定化，不做隐式转换

短期不要尝试把 P1S/Bambu G-code 自动转换成 U1 G-code。启动段、换料、擦嘴、
校准、runout、相机和恢复逻辑都与机型深度耦合，做字符串级转换风险很高。

正确策略：

```text
非 U1 G-code：拒绝
U1 G-code + native/ACE 映射明确：允许进入 rewrite
rewrite 后再次校验：通过才上传
```

### 2. rewrite 后必须二次校验

preflight 原始文件通过校验不够。后端 rewrite / inject auto-load 后，应对最终
将上传的 G-code 再跑一次轻量校验。

重点检查：

- 是否引入了新的危险命令。
- `ACE_SWAP_HEAD` 是否只作用在配置为 ACE 的 head 上。
- 初始 auto-load 是否插入在安全 anchor 后。
- 最终文件里 UI 展示的 source map 与真实命令一致。

### 3. auto-load anchor 要绑定实机安全状态

当前 auto-load 目标是“在首个需要 ACE 耗材的位置前加载”。实机上还必须满足：

```text
XY/Z home 状态明确
目标 head 合法
不会打断原厂启动段的擦嘴/校准状态机
不会在 runout 检测打开前后制造假报警
```

后续优化应把 anchor 从“文本位置优先级”升级为“状态条件优先级”：

```text
已完成必要 home
已完成原厂 wipe / bed leveling / start-line 前的安全准备
即将进入首个真实挤出段
```

### 4. 建立实机回归样本库

每次实机事故都应保存最小证据：

```text
原始 G-code header + 前 500 条命令
最终上传 G-code header + 前 500 条命令
source_map
route_plan
Klipper 报错上下文
Moonraker 上传记录
```

样本分类：

```text
valid_u1_native_only
valid_u1_single_ace
valid_u1_native_plus_ace
invalid_bambu_p1s
invalid_unknown_machine
invalid_dangerous_start_gcode
```

这些样本应进入 Docker dry-run 或离线单元测试，避免后续改 preflight 时回退。

### 5. 官方发送路径差异需要专项记录

后续要对比官方软件发送和 Web 发送的差异：

```text
上传 API
print=true 触发方式
打印任务配置文件
是否写入 snapmaker/print_task*.json
是否先调用 SET_PRINT_PREFERENCES
是否有官方后处理
是否使用不同 gcodes root 或 metadata
```

不要假设“同一份 G-code 官方能打，Web 也应该能打”。官方路径可能附带了额外
任务状态、机型适配和安全检查。
