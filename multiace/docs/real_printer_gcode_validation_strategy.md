# 实机 G-code 校验与 Web 发送策略

日期：2026-06-06

背景：在 Snapmaker U1 实机 `192.168.1.38` 上，用户通过
`http://192.168.1.38/multiace/` Web 页面上传并发送打印时，连续触发
Z 轴相关报错；同类任务通过官方切片软件直接发送可以正常打印。

本文记录本次问题的定位结论，并把后续必须落地的 Web 校验规则和实机优化策略
固化为工程要求。

## 实机现象

最近失败集中在同一份文件：

```text
六边拉桥_PLA_39m12s.gcode
```

Klipper 日志中最近三次失败时间：

```text
2026-06-06 14:14:35
2026-06-06 14:15:17
2026-06-06 14:19:28
```

核心报错一致：

```text
Must home Z axis first: 227.200 300.500 205.020
```

失败行：

```text
current_line_gcode=G380 S2 Z-25 F300 ;
```

报错前还出现以下 U1 Klipper 当前不支持或不兼容的命令：

```text
Unknown command:"M486"
Unknown command:"M201"
Unknown command:"M203"
Unknown command:"M205"
Unknown command:"M710"
Unknown command:"M290"
Unknown command:"M17"
Unknown command:"G380"
```

上传文件头部显示：

```gcode
; generated by Snapmaker Orca 2.3.1
;===== machine: P1S ========================
```

结论：Web 发送路径接受并直接打印了一份 Bambu/P1S 风格 G-code。该启动段不适配
U1 的 Klipper 环境，导致在 Z 未正确归零前执行 `G380` 相关动作并触发 hard
error。

## 根因判断

这不是 Z 轴硬件故障，也不是普通打印路径中的 Z 移动越界。

根因是 Web preflight 当前只关注 native/ACE 映射和 G-code rewrite，没有对
“这份 G-code 是否属于当前 U1 机型/当前固件可执行方言”做硬性校验。官方软件
直接发送正常，是因为它使用官方 U1 发送路径，或者在发送前完成了机型模板、
打印任务参数和固件命令集适配。

Web 路径当前行为：

```text
用户选择 G-code
-> /api/preflight
-> native/ACE source map
-> rewrite / inject auto-load
-> Moonraker /server/files/upload print=true
```

缺失的安全门：

```text
G-code 机型签名校验
G-code 方言/危险命令校验
启动段 home 顺序校验
官方任务参数与 Web 发送参数差异校验
```

## 必须新增的校验规则

### 1. 机型签名 blocker

Web preflight 必须扫描文件头部和 slicer 元数据。发现明显非 U1 目标机型时，
直接拒绝发送打印。

第一批 hard blocker：

```text
;===== machine: P1S
;===== machine: X1
;===== machine: X1C
;===== machine: A1
;===== machine: A1 mini
Bambu
```

错误提示应明确：

```text
当前 G-code 不是 Snapmaker U1 兼容文件。请使用 U1 配置重新切片后再通过
multiACE Web 上传。
```

### 2. 危险命令 blocker

发现以下命令时，默认拒绝 Web 发送。后续若确认某些命令在 U1 固件上安全，
再从 blocker 降级为 warning。

```text
G380
M710
M290
M960
M975
M970
M974
M620
M621
M620.1
M620.11
M628
M629
```

理由：

- `G380` 在本次事故中直接触发 `Must home Z axis first`。
- `M620/M621/M620.*` 是 AMS/Bambu 风格换料语义，不应在 U1 Web 发送路径中
  裸执行。
- `M970/M974/M975/M960` 属于机型特定检测/灯光/振动相关命令，不能假定 U1
  Klipper 支持。

### 3. 启动段 home 顺序校验

Web preflight 需要检查执行块前段，至少覆盖前 500 条非注释 G-code。

hard blocker 条件：

```text
在 G28 或 G28 Z 之前出现绝对 Z 移动
在 Z 未 homed 前出现 G38/G380/探针/校准类 Z 动作
在明显未完成 home 的状态下出现负 Y 擦嘴/丢料路径
```

本次样例中的失败模式：

```gcode
G91
G380 S2 Z30 F300
G380 S2 Z-25 F300
```

这类逻辑即使在源机型上合法，也不能直接交给 U1 Web 路径。

### 4. 已知无害 unknown command 不应掩盖 blocker

某些命令如 `M486` 可能只是对象标记，单独出现时可以 warning。但是只要同一
文件出现机型签名 blocker 或危险命令 blocker，就必须拒绝发送，不能因为
“Unknown command 可忽略”而继续打印。

### 5. 校验结果进入 source map

每次 preflight 的 `source_map` 或旁路 artifact 应记录：

```text
validation.version
validation.machine_signature
validation.blockers[]
validation.warnings[]
validation.scanned_header_lines
validation.scanned_command_lines
```

这样实机日志只看到 Klipper 报错时，也能回溯 Web 在发送前看到过什么。

## Web UI 行为要求

校验失败时：

- 禁用发送打印按钮。
- 显示第一个 hard blocker 的人类可读原因。
- 展开详情时显示命令、行号、规则名。
- 不提供“忽略并打印”按钮，除非后续实现专门的开发者调试模式。

示例：

```text
无法发送打印：检测到 P1S/Bambu 启动 G-code，不兼容 Snapmaker U1 Web 发送路径。
第 42 行：;===== machine: P1S
第 67 行：G380 S2 Z-25 F300
请使用 U1 配置重新切片。
```

## 实机优化策略

### 1. Web 发送路径先做稳定化，不做隐式转换

短期不要尝试把 P1S/Bambu G-code 自动转换成 U1 G-code。启动段、换料、擦嘴、
校准、runout、相机和恢复逻辑都与机型深度耦合，做字符串级转换风险很高。

正确策略：

```text
非 U1 G-code：拒绝
U1 G-code + native/ACE 映射明确：允许进入 rewrite
rewrite 后再次校验：通过才上传
```

### 2. rewrite 后必须二次校验

preflight 原始文件通过校验不够。后端 rewrite / inject auto-load 后，应对最终
将上传的 G-code 再跑一次轻量校验。

重点检查：

- 是否引入了新的危险命令。
- `ACE_SWAP_HEAD` 是否只作用在配置为 ACE 的 head 上。
- 初始 auto-load 是否插入在安全 anchor 后。
- 最终文件里 UI 展示的 source map 与真实命令一致。

### 3. auto-load anchor 要绑定实机安全状态

当前 auto-load 目标是“在首个需要 ACE 耗材的位置前加载”。实机上还必须满足：

```text
XY/Z home 状态明确
目标 head 合法
不会打断原厂启动段的擦嘴/校准状态机
不会在 runout 检测打开前后制造假报警
```

后续优化应把 anchor 从“文本位置优先级”升级为“状态条件优先级”：

```text
已完成必要 home
已完成原厂 wipe / bed leveling / start-line 前的安全准备
即将进入首个真实挤出段
```

### 4. 建立实机回归样本库

每次实机事故都应保存最小证据：

```text
原始 G-code header + 前 500 条命令
最终上传 G-code header + 前 500 条命令
source_map
route_plan
Klipper 报错上下文
Moonraker 上传记录
```

样本分类：

```text
valid_u1_native_only
valid_u1_single_ace
valid_u1_native_plus_ace
invalid_bambu_p1s
invalid_unknown_machine
invalid_dangerous_start_gcode
```

这些样本应进入 Docker dry-run 或离线单元测试，避免后续改 preflight 时回退。

### 5. 官方发送路径差异需要专项记录

后续要对比官方软件发送和 Web 发送的差异：

```text
上传 API
print=true 触发方式
打印任务配置文件
是否写入 snapmaker/print_task*.json
是否先调用 SET_PRINT_PREFERENCES
是否有官方后处理
是否使用不同 gcodes root 或 metadata
```

不要假设“同一份 G-code 官方能打，Web 也应该能打”。官方路径可能附带了额外
任务状态、机型适配和安全检查。

## 验收标准

第一阶段完成标准：

- P1S/Bambu 样例在 Web preflight 阶段被拒绝，不再进入 Moonraker upload。已完成。
- 错误提示包含机型不兼容、危险命令、行号。已完成。
- 合法 U1 native-only 文件不被误拒。已由 dry-run 覆盖。
- 合法 U1 single ACE 文件仍能生成 source map 和 route plan。已由 dry-run 覆盖。
- rewrite 后最终上传文件能通过二次校验。已完成。
- dry-run 覆盖至少一个 `invalid_bambu_p1s` 样例。已完成。

- `/api/preflight` 在保存 preflight artifact 前扫描原始上传文件。
